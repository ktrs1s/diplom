# Тест-план EXCLUSIVE

## 1. Цель

Проверить, что интернет-магазин EXCLUSIVE стабильно работает для клиента, администратора и менеджера Telegram-бота, корректно обрабатывает каталог, корзину, авторизацию, заказы и хранение данных.

## 2. Объект тестирования

В проект входят:

- витрина сайта;
- каталог и карточки товаров;
- корзина и оформление заказа;
- авторизация и личный кабинет;
- админка каталога;
- backend API;
- PostgreSQL;
- Telegram-бот;
- интеграция с Яндекс.Метрикой.

## 3. Технологический контур

- frontend: `HTML`, `CSS`, `JavaScript`
- backend: `Python`
- база данных: `PostgreSQL`
- веб-сервер: `nginx`
- контейнеризация: `Docker`, `docker compose`
- внешняя интеграция: `Telegram Bot API`, `Yandex Metrika`

## 4. Область тестирования

### Входит в тестирование

- шапка и подвал на всех страницах;
- главная страница;
- каталог;
- карточка товара;
- корзина;
- регистрация и вход;
- личный кабинет;
- админка;
- оформление заказа через Telegram;
- API и база данных;
- адаптивность;
- базовая аналитика.

### Не входит в этот тест-план

- нагрузочное тестирование высокого уровня;
- отказоустойчивость инфраструктуры хостинга;
- платежные шлюзы;
- интеграции со сторонними CRM.

## 5. Типы тестирования

- smoke-тестирование после деплоя;
- функциональное тестирование;
- интеграционное тестирование;
- регрессионное тестирование;
- UI/UX тестирование;
- адаптивное тестирование;
- кроссбраузерное тестирование;
- базовое тестирование безопасности и прав доступа;
- проверка данных в БД.

## 6. Среды тестирования

### Основная среда

- локальный запуск через `docker compose`
- сайт: `http://localhost:8080`
- API: `http://localhost:9000/api`

### Целевая среда

- домен продакшна: `exclusivegu.ru`

### Браузеры

- `Safari`
- `Chrome`
- `Yandex Browser`

### Устройства

- desktop от `1440px`
- tablet `768px–1024px`
- mobile `320px–430px`

## 7. Роли пользователей

- гость;
- зарегистрированный пользователь;
- администратор;
- менеджер Telegram-бота.

## 8. Критические бизнес-сценарии

1. Пользователь открывает сайт, переходит в каталог и находит товар.
2. Пользователь открывает карточку товара, выбирает размер и цвет, добавляет в корзину.
3. Пользователь открывает корзину, авторизуется по номеру телефона и уходит в Telegram для продолжения заказа.
4. Бот принимает `/start order_...`, прикрепляет чат к заказу и предлагает подтвердить параметры заказа.
5. Менеджер получает заказ в Telegram и меняет статус.
6. Пользователь видит завершённый заказ в личном кабинете.
7. Администратор входит под своим номером, открывает админку, меняет товар или категорию, и изменения появляются на витрине.

## 9. Критерии начала тестирования

- контейнеры `web`, `api`, `postgres` подняты;
- backend отвечает на `/api/health`;
- база данных доступна;
- Telegram-бот имеет валидный токен;
- для админки известен админский номер;
- каталог успешно загружается из API.

## 10. Критерии завершения тестирования

- пройдены все smoke-сценарии;
- пройдены все сценарии с приоритетом `P1`;
- нет критических и блокирующих дефектов;
- нет дефектов, мешающих оформлению заказа;
- нет дефектов, нарушающих права доступа между обычным пользователем и администратором.

## 11. Приоритеты

- `P0`: сайт не открывается, API недоступно, бот не отвечает, заказ не создаётся
- `P1`: нельзя войти, нельзя добавить товар в корзину, нельзя сохранить товар в админке
- `P2`: ошибки отображения, адаптива, текста, аналитики
- `P3`: косметические проблемы

## 12. Тестовые данные

### Пользователи

- демо-номер: `+7 (999) 999-99-99`
- админский номер: `+7 (995) 398-02-43`
- тестовый пользователь без заказов
- тестовый пользователь с завершённым заказом

### Данные каталога

- категория с товарами;
- категория без товаров;
- товар с одним размером;
- товар с несколькими размерами;
- товар без изображения;
- товар с пользовательским изображением.

### Заказы

- черновик заказа;
- заказ со статусом `pending_manager`;
- завершённый заказ;
- отменённый заказ.

## 13. Smoke-чеклист

- открывается `/`
- открывается `/catalog/`
- открывается карточка товара
- работает добавление в корзину
- открывается `/cart/`
- работает переход на `/auth/`
- работает вход по номеру
- открывается `/account/`
- для админа открывается `/admin/`
- создаётся заказ через корзину
- бот отвечает на `/start`
- API `/api/catalog` и `/api/health` возвращают `200`

## 14. Подробные тестовые сценарии

### 14.1. Общая навигация, шапка и подвал

#### Цель

Проверить единообразие шапки и подвала на всех страницах.

#### Проверки

- логотип ведёт на главную;
- в шапке отсутствует пункт `Блог`;
- активный пункт меню корректно подсвечивается;
- при наведении на пункты меню появляется плавное подчёркивание слева направо;
- сетка шапки одинакова на главной, каталоге, товаре, корзине, авторизации, кабинете и админке;
- иконка профиля ведёт на страницу входа или кабинет;
- иконка корзины ведёт в корзину;
- footer одинаковый на всех страницах;
- в footer нет ссылок `Условия обмена и возврата`, `Доставка и оплата`, `Хиты продаж`, `VK`, `Блог`;
- присутствует подпись `Meta запрещена на территории России`.

### 14.2. Главная страница

#### Проверки

- страница открывается по `/`;
- в адресной строке нет `/главная%20страница/index.html`;
- основной баннер отображается корректно;
- секции категорий и подборок открываются без наложений и разрывов;
- карточки товара отображаются без цветных кружков под товаром;
- удалён Telegram-баннер внизу главной;
- клики по карточкам ведут на товар или каталог;
- на desktop видно ожидаемое число карточек в каруселях;
- на tablet и mobile блоки не съезжают и не дают горизонтальный скролл.

### 14.3. Каталог

#### Проверки

- страница открывается по `/catalog/`;
- хлебные крошки отображаются корректно;
- наверху остаётся только заголовок `Каталог`, если так задумано по текущему дизайну;
- шапка идентична главной;
- категории слева открываются корректно;
- по умолчанию не показывается “змейка” из всех товаров в админке, но в каталоге карточки выводятся сеткой;
- карточки товаров не имеют лишних рамок вокруг изображения;
- фильтрация по разделу работает;
- карточка без изображения показывает заглушку;
- цена, название, размеры отображаются корректно;
- URL категории работает через чистый адрес.

### 14.4. Карточка товара

#### Проверки

- страница товара открывается по чистому URL;
- хлебные крошки корректны;
- галерея изображений работает;
- у миниатюр нет лишних рамок;
- выбор размера работает;
- выбор цвета работает;
- кнопка по умолчанию `Добавить в корзину`;
- после клика текст меняется на `Добавлено в корзину`;
- после добавления нет редиректа в корзину;
- переход в корзину происходит только по иконке в шапке;
- отображаются состав, характеристики и уход;
- блок похожих товаров отображается без лишних внутренних рамок.

### 14.5. Корзина

#### Проверки

- страница открывается по `/cart/`;
- пустая корзина центрирована и выглядит аккуратно;
- кнопка `Перейти в каталог` маленькая и по центру;
- лишние блоки, статистика и сервисные карточки отсутствуют;
- при добавлении товара пустое состояние скрывается;
- товар отображается с названием, ценой, размером, цветом и количеством;
- можно изменить количество;
- можно удалить товар;
- сумма, доставка и итог считаются корректно;
- бесплатная доставка работает по порогу;
- данные корзины сохраняются между страницами;
- счётчик корзины в шапке обновляется.

### 14.6. Авторизация и регистрация

#### Проверки

- страница открывается по `/auth/`;
- шапка и footer совпадают с остальными страницами;
- экран авторизации не содержит лишнего текста и лишних блоков;
- номер телефона маскируется как `+7 (999) 999-99-99`;
- `+7` подставляется сразу;
- цифры в скобках редактируются корректно;
- вход возможен по существующему номеру;
- регистрация требует только имя, фамилию и телефон;
- после входа открывается личный кабинет;
- при полном отсутствии активности более 15 минут требуется новый вход;
- при открытой активной сессии повторный вход не требуется.

### 14.7. Личный кабинет

#### Проверки

- страница открывается по `/account/` только для авторизованного пользователя;
- для неавторизованного пользователя идёт редирект на вход;
- отображаются имя, фамилия и телефон;
- можно изменить имя;
- можно изменить фамилию;
- можно изменить телефон;
- после сохранения появляется сообщение `Успешно.`;
- сообщение исчезает автоматически;
- обычный пользователь не видит кнопку открытия админки;
- отображаются только завершённые заказы;
- заказ выглядит как карточка с датой, суммой и миниатюрами товаров;
- незавершённые или черновые заказы не показываются.

### 14.8. Админка

#### Проверки

- страница открывается по `/admin/` только для администратора;
- обычный пользователь доступа не имеет;
- шапка стилистически совпадает с главной страницей;
- интерфейс белый, простой и не перегруженный;
- сначала выбирается категория;
- товары не показываются всей длинной лентой до выбора категории;
- после выбора категории открывается список товаров этой категории;
- можно выбрать конкретный товар для редактирования;
- можно изменить:
  - название;
  - цену;
  - категорию;
  - размеры;
  - изображение;
  - состав;
  - характеристики;
  - уход;
- можно добавить новую категорию;
- можно добавить новый товар;
- новый товар появляется в начале списка;
- изменения сохраняются в БД и отображаются на витрине;
- загрузка изображения работает;
- при ошибке загрузки есть понятная реакция интерфейса.

### 14.9. Оформление заказа и Telegram-бот

#### Проверки

- из корзины нельзя начать checkout с пустой корзиной;
- неавторизованный пользователь перед checkout уходит на авторизацию;
- авторизованный пользователь создаёт заказ;
- сайт формирует корректный deep link в Telegram;
- бот открывается по ссылке `order_<id>`;
- команда `/start order_<id>` привязывает чат к заказу;
- бот показывает состав заказа и сумму;
- пользователь может выбрать тип доставки;
- пользователь может добавить комментарий;
- пользователь может подтвердить заказ;
- заказ получает статус `pending_manager`;
- менеджеру приходит уведомление;
- менеджер может менять статусы заказа;
- пользователь получает обновление статуса;
- при рестарте backend бот продолжает polling;
- pending updates не зависают в Telegram.

### 14.10. API

#### Проверки

- `GET /api/health`
- `GET /api/catalog`
- `GET /api/config`
- `POST /api/login`
- `POST /api/register`
- `POST /api/orders`
- `POST /api/catalog/save`
- ответы имеют корректный HTTP status;
- ошибки возвращаются в JSON;
- невалидные данные не приводят к падению backend;
- неавторизованный доступ к admin API отклоняется.

### 14.11. База данных

#### Проверки

- товары сохраняются в `PostgreSQL`;
- у каждого товара есть уникальный `id`;
- пользователь хранится с именем, фамилией и телефоном;
- заказ хранится вместе с составом заказа;
- изменение товара в админке отражается в БД;
- новый пользователь после регистрации появляется в БД;
- завершённый заказ доступен для кабинета пользователя;
- данные после перезапуска контейнеров не теряются.

### 14.12. Яндекс.Метрика

#### Проверки

- счётчик подключён на всех основных страницах;
- код Метрики не дублируется;
- `noscript`-блок присутствует;
- переходы между ключевыми страницами не ломают загрузку Метрики;
- при необходимости события ecommerce можно расширить отдельным пакетом тестов.

### 14.13. Адаптивность

#### Проверки

- на desktop нет визуальных разрывов;
- на tablet блоки перестраиваются корректно;
- на mobile нет горизонтального скролла;
- кнопки доступны для нажатия пальцем;
- карточки товаров не ломаются;
- корзина адаптивна;
- авторизация адаптивна;
- админка пригодна хотя бы для базового использования на tablet.

### 14.14. Кроссбраузерность

#### Проверки

- `Safari` desktop;
- `Chrome` desktop;
- `Yandex Browser` desktop;
- `Safari` mobile;
- `Chrome` mobile на Android, если есть доступ;
- маска телефона одинаково работает в разных браузерах;
- Telegram deep link не ломается в браузерах.

## 15. Нефункциональные проверки

- страницы не должны заметно мигать при загрузке;
- изображения должны подгружаться без битых ссылок;
- в консоли браузера не должно быть критических ошибок;
- backend не должен падать при обычных пользовательских действиях;
- после `docker compose up -d --build` сайт должен быть готов к smoke-проверке.

## 16. Риски

- потеря связи между сайтом и Telegram-ботом;
- ошибки сохранения каталога из админки;
- рассинхронизация local storage и backend API;
- ошибки сессии после простоя;
- битые изображения или неверные пути к файлам;
- различия в работе маски телефона между браузерами;
- проблемы прав доступа между обычным пользователем и администратором.

## 17. Рекомендуемый порядок регресса перед релизом

1. Проверить старт контейнеров.
2. Пройти smoke по главной, каталогу, товару и корзине.
3. Проверить вход, регистрацию и кабинет.
4. Проверить создание заказа и переход в Telegram.
5. Проверить получение заказа менеджером и смену статуса.
6. Проверить админку: изменение товара, сохранение, появление на витрине.
7. Проверить адаптив на mobile и tablet.
8. Проверить Метрику и консоль браузера.

- дата и версия;
- среда;
- кто тестировал;
- список пройденных сценариев;
- список дефектов;
- итоговый статус: `pass`, `pass with issues`, `fail`.
